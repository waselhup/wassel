# A16 — Discovery is private; vendor names are invisible

**Date:** 2026-05-25
**Status:** Accepted
**Rule:** R16

## Context

LinkedIn data is ingested for Radar via a third-party vendor (Apify, `harvestapi/linkedin-profile-search`). The vendor relationship is operational; it is not part of the product story to users.

## Decision

In all user-facing surfaces, the operation is called **"اكتشاف"** in Arabic and **"discovery"** in English. The word **"Apify"** never appears. Neither do "Bright Data", "Apollo", "Waalaxy", or any other vendor.

The user's discovered LinkedIn data is private to the user. Another user cannot see, search, or browse it. RLS in Supabase enforces this at the database layer.

If Wassel changes vendors later (e.g. moves from Apify to a Bright Data fallback), users see no change.

## Consequences

- All UI strings, prompts, marketing copy, and errors use "اكتشاف" / "discovery"
- `linkedin_data` and `profile_analyses` tables have strict RLS: `auth.uid() = user_id`
- Vendor names live only in environment variables and server logs

## Alternatives Rejected

- "Powered by Apify" badge — rejected, contradicts the product story
- Public profile sharing — out of scope (revisit in Sprint 8+)
