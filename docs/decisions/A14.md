# A14 — The activity log is the single suggestions source

**Date:** 2026-05-25
**Status:** Accepted
**Rule:** R14

## Context

Without discipline, a dashboard accumulates hand-coded rules: "if user hasn't logged in for 7 days, show X", "if user has 5 incomplete drafts, show Y". This grows unmaintainable.

## Decision

There is one source for AI-driven suggestions: the `activity_log` table. The `next-task` prompt is the only function that reads it for suggestion-generation purposes. No screen invents activity it did not observe.

`activity_log` rows are short:
```
(id, user_id, action, target, payload JSON, created_at)
```

Actions are documented in `docs/PRD/06-errors-roadmap.md` (the same doc covers the activity catalogue for now; Sprint 6 may split it out).

## Consequences

- Every router that performs a meaningful user action writes one `activity_log` row
- The activity log is the *event source*, not a join target — keep payloads small
- AI suggestions live in `ai_suggestions`, derived nightly; this is the only path from log → UI

## Alternatives Rejected

- Live computation in the Dashboard endpoint — rejected, latency
- Hand-rolled rules — rejected, doesn't scale and contradicts R14
