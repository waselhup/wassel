# A06 — Errors speak the user's language

**Date:** 2026-05-25
**Status:** Accepted
**Rule:** R06

## Context

Pre-Copilot error messages leaked runtime details ("HTTP 429 from upstream", "timeout after 60000 ms"). Users saw English stack traces in an Arabic UI.

## Decision

All user-facing errors are:
- Translated through the `errors.*` namespace in `translation.json`
- Plain language: what happened + what the user can do
- Free of runtime brand or vendor names
- Always include the wallet reassurance ("لم نخصم أي توكن") if a spend could have been involved

Sprint 8 adds an "Error Formatter" that takes a raw exception and rewrites it into the standard voice using `claude-haiku-4-5-20251001` and the `error-formatter` prompt. Until then, manual mappings live in `errors.*`.

## Consequences

- A new `errors.*` namespace in both `ar` and `en` translation files
- The shared API error envelope (`{ code, messageKey, params }`) replaces raw `message`
- Sentry still captures the raw exception for engineering; the user sees the formatted message

## Alternatives Rejected

- Show the raw error in dev only — rejected, half-measure
- One-size-fits-all "Something went wrong" — rejected, gives the user no agency
