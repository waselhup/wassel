# A02 — The user is never asked the same question twice

**Date:** 2026-05-25
**Status:** Accepted
**Rule:** R02

## Context

Pre-Copilot Wassel asked for the user's goal, target role, industry, and language **in every flow**. The CV Builder didn't know what the Radar had learned. Users would write "Senior PM at Fintech" three times in one session.

## Decision

Introduce `career_profile` as the single source of truth. Every screen that needs goal/level/target_role/industry/language reads from this table. Temporary divergences ("try the Radar as if I were targeting Director instead") write to `section_overrides`, which decay (default 24h) and never mutate `career_profile`.

If a screen wants to ask the user one of these fields again, the screen is wrong, not the rule.

## Consequences

- A new `career_profile` table, written once during onboarding
- A new `section_overrides` table for short-lived experiments
- Every router that previously asked for goal/role now reads from these
- A `careerProfile` tRPC router exposes `get / create / update / delete / export / sectionOverride`
- Settings → Career Profile is the only place these fields can change

## Alternatives Rejected

- Cache the answers in localStorage — rejected, can't survive device switches
- Put the answers into the JWT — rejected, can't update without sign-out
