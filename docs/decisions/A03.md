# A03 — Tokens are deducted only on success

**Date:** 2026-05-25
**Status:** Accepted
**Rule:** R03

## Context

Earlier Wassel code deducted tokens before calling the model. If the model errored, timed out, or the user disconnected, the user was charged for nothing.

## Decision

Token deduction is the **last** step of a successful operation, not the first. The flow is:

1. Pre-check: does the user have enough tokens across `wallet_bonus + wallet_subscription + wallet_topup`?
2. Reserve (optional, atomic) — record an intent in `wallet_transactions` with status `pending`
3. Execute the operation
4. **On success:** atomic deduction via `deduct_tokens_v2`, mark the transaction `committed`
5. **On failure:** rollback — no balance changes, mark the reservation `failed`

The reservation step prevents two parallel requests from both passing the pre-check, but only one would have enough tokens to complete. The reservation is optional for cheap ops (e.g. cache hits = 0 tokens).

## Consequences

- All token-spending routers wrap the model call in a try/catch and only call `deduct_tokens_v2` in the success branch
- `wallet_transactions` carries a status field: `pending | committed | failed | refunded`
- Error messages include "لم نخصم أي توكن" / "No tokens were spent" when relevant

## Alternatives Rejected

- Pre-deduct + refund on failure — rejected, refunds are hard to audit and erode trust
- Skip reservation entirely — rejected, allows a single user to double-spend via parallel tabs
