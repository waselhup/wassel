# A15 — Users own their data (PDPL compliance)

**Date:** 2026-05-25
**Status:** Accepted
**Rule:** R15

## Context

Saudi PDPL (Personal Data Protection Law, enforced 2024) requires:
- The user can view all data held about them
- The user can request export in a usable format
- The user can request deletion of all personal data

## Decision

Settings → Privacy & Data exposes two buttons:

1. **"تحميل بياناتي"** / "Download my data" — calls `careerProfile.export()`, returns a JSON blob with everything Wassel stores for the user (career_profile, overrides, transactions, activity log, all artifacts)
2. **"حذف كل بياناتي نهائياً"** / "Delete everything permanently" — a danger button that requires a typed confirmation. Calls `careerProfile.delete()`, which soft-deletes the auth row and hard-deletes user-owned tables in a Supabase trigger.

PDPL also requires showing the user *how* their data is used. The Privacy page includes:
- "كيف نستخدم بياناتك" — short plain-language explanation
- "مصادر التحليل المُستخدمة + آخر تحديث" — sources for Radar / discovery data and their refresh times

## Consequences

- `careerProfile.export()` and `careerProfile.delete()` ship in Sprint 2
- A Supabase trigger on the auth.users soft-delete cascades to user-owned tables
- The Privacy page is required-by-law content, never gated behind a feature flag

## Alternatives Rejected

- Export-by-email-request workflow — rejected, slower than self-service
- Hide the delete button to reduce churn — rejected, violates PDPL
