# A09 — Caching is non-negotiable

**Date:** 2026-05-25
**Status:** Accepted
**Rule:** R09

## Context

Re-rendering a Radar result for the same profile and target role should cost zero wallet tokens. Users who delete and re-open results should not be billed twice.

## Decision

Every model call is keyed and cached in Supabase tables (`radar_cache`, `resume_cache`, `content_cache`). Cache keys:

| Surface | Key | TTL |
| --- | --- | --- |
| Radar | `(user_id, target_role, profile_hash)` | until inputs change |
| Resume | `(user_id, target_role, profile_hash, version)` | until inputs change |
| Content post | `(user_id, topic_hash, language)` | 7 days |
| Carousel | `(user_id, topic_hash, language)` | 7 days |
| Next-task | `(user_id, date)` | 24h |

`profile_hash = sha256(JSON.stringify(career_profile + manual_overrides))`.

A cache hit returns the stored artifact and records a `cache_hit` row in `activity_log` for analytics. **It does NOT call `deduct_tokens_v2`.**

## Consequences

- Three new cache tables (added in Sprint 3 / 4 / 5)
- A `getOrCache(key, fn)` helper in `server/_core/lib/cache.ts`
- The "Run again" button on a Radar result is replaced by "نسخة جديدة" which forces cache miss (R-named per R04)

## Alternatives Rejected

- In-memory cache — rejected, Vercel functions are ephemeral
- 1-hour TTL globally — rejected, doesn't match input-driven invalidation
